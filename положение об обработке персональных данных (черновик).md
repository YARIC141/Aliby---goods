## Раздел 4. Порядок сбора и хранения персональных данных

**4.1.** Персональные данные собираются следующими способами:
- непосредственно от субъекта при регистрации и использовании Сервиса;
- автоматически в ходе технического взаимодействия субъекта с Сервисом (cookie, IP, user-agent).

**4.2.** Все персональные данные хранятся на серверах в Российской Федерации (г. Москва, дата-центр Selectel). Оператор не допускает хранения ПД за пределами РФ, за исключением FCM-токена, обрабатываемого Google LLC (США) исключительно в целях доставки push-уведомлений (см. раздел 7).

**4.3.** Доступ к персональным данным в базе данных ограничен на уровне строк (Row Level Security, Supabase/PostgreSQL): каждый пользователь видит только свои данные. Доступ к административным функциям БД имеет только Оператор.

**4.4.** Пароли пользователей не хранятся в открытом виде. Хранится только криптографический хэш (bcrypt) средствами системы аутентификации Supabase Auth.

**4.5.** API-ключи банка-эквайера шифруются при хранении в базе данных.

## Раздел 5. Сроки хранения и порядок уничтожения

| Категория данных | Срок хранения | Основание |
|-----------------|--------------|-----------|
| Профиль, история заказов | До удаления аккаунта или 3 года с последней активности | Договор с субъектом |
| Данные электронной подписи договора (хэш, timestamp) | Срок исковой давности — 3 года с даты подписания, но не менее 5 лет | ГК РФ ст. 196, ФЗ-63 |
| Данные транзакций (без платёжных реквизитов карты) | 5 лет | ФЗ-402 «О бухгалтерском учёте» |
| Технические логи (IP, user-agent) | 1 год | Технические нужды |
| FCM-токен | До отзыва согласия или удаления аккаунта | Согласие субъекта |

**5.1. Порядок уничтожения данных при удалении аккаунта:**

При нажатии кнопки «Удалить аккаунт» в приложении Оператор:
1. Направляет запрос на удаление в Supabase Auth Edge Function `delete-account`.
2. В течение 30 дней удаляются: профиль, адрес, FCM-токен, история заказов и бронирований (если не требуют хранения по законодательству).
3. Данные электронной подписи договора и транзакционная история сохраняются в течение установленных законом сроков в анонимизированной форме.

> **Требует уточнения у юриста:** достаточна ли анонимизация транзакционной истории или необходимо полное удаление.

**5.2.** По истечении 3 лет с даты последней активности неактивные аккаунты подлежат удалению в автоматическом режиме. Реализация автоматического удаления — в плане разработки.

## Раздел 7. Передача персональных данных третьим лицам

**7.1. Банк-эквайер Продавца**  
При оформлении заказа с онлайн-оплатой Покупатель переходит на защищённую страницу банка-эквайера, самостоятельно выбранного Продавцом. Оператор Платформы не передаёт ПД Покупателя в банк — Покупатель вводит платёжные данные непосредственно на странице банка. Банк получает от Платформы только технические параметры транзакции (сумма, идентификатор заказа). Банк является самостоятельным оператором ПД в части платёжной операции.

**7.2. Google LLC (Firebase Cloud Messaging) — трансграничная передача**  
- **Что передаётся:** уникальный FCM-токен устройства пользователя и текст уведомления (статус заказа).
- **Куда:** серверы Google LLC, США (страна, не включённая в перечень государств с адекватной защитой ПД по Приказу РКН).
- **Правовое основание:** ст. 12, ч. 4, п. 2 ФЗ-152 — передача необходима для исполнения договора с субъектом.
- **Как отказаться:** пользователь отключает уведомления в настройках устройства.
- **Обязательства:** Оператор уведомляет РКН о трансграничной передаче в установленном порядке (статус: требует подачи уведомления).

**7.3. Государственные органы**  
ПД могут быть переданы по запросу уполномоченных органов РФ в порядке, предусмотренном действующим законодательством.

**7.4.** Иная передача персональных данных третьим лицам не осуществляется. Продажа и обмен ПД в коммерческих целях запрещены.

## Раздел 8. Меры по обеспечению безопасности персональных данных

**8.1. Технические меры:**

| Мера | Реализация |
|------|-----------|
| Шифрование канала связи | HTTPS/TLS 1.2+ через Cloudflare на всех эндпоинтах |
| Разграничение доступа к данным | Row Level Security (PostgreSQL/Supabase) — пользователь видит только свои данные |
| Защита от перебора | Nginx rate limiting, Kong rate-limit plugin |
| Брандмауэр | UFW, SSH только с ключом на нестандартном порту |
| Хранение паролей | bcrypt (Supabase Auth), пароли в открытом виде не хранятся |
| Аудит действий | Функции аудита в БД (статус: проверить включение в prod) |
| Бэкапы | Автоматические резервные копии БД (проверить расписание и хранение) |

**8.2. Организационные меры:**

- Доступ к производственной базе данных имеет только Оператор (ИП Копылов Я.Д.).
- Сотрудники (роль «employee») имеют ограниченный доступ только к функциям QR-сканера и просмотра заказов без доступа к ПД других пользователей.
- При добавлении новых разработчиков или подрядчиков — оформление соглашения о конфиденциальности обязательно.

**8.3. Уровень защищённости:**

Определяется по ПП РФ №1119 от 01.11.2012 «Об утверждении требований к защите персональных данных при их обработке в информационных системах персональных данных».

> **Требует определения юристом/специалистом по ИБ:** уровень защищённости ИСПДн (УЗ-1..УЗ-4) и перечень необходимых мер по соответствующему приказу ФСТЭК.  
> Предварительная оценка: УЗ-3 (обычные ПД, не специальные, до 100 000 субъектов) — не требует аттестации, но требует ряда организационных мер.

## Раздел 11. Оценка вреда субъектам персональных данных

> **Требование:** ФЗ-152 ст. 18.1, п. 4 — оператор обязан оценивать вред, который может быть причинён субъектам ПД при нарушении требований Закона.

| Сценарий нарушения | Вероятный вред субъектам | Оценка риска | Меры снижения |
|-------------------|------------------------|-------------|--------------|
| Утечка email и имени | Спам, фишинг | Средний | HTTPS, rate limiting, RLS |
| Утечка адреса доставки и геопозиции | Установление места жительства, физическая угроза | Высокий | RLS, доступ только самому субъекту |
| Утечка истории заказов | Раскрытие пищевых предпочтений, распорядка дня | Средний | RLS |
| Утечка FCM-токена | Отправка вредоносных уведомлений | Низкий | Токен действует до сброса устройства |
| Утечка API-ключей банка Продавца | Финансовые потери Продавца | Критический | Шифрование при хранении, доступ только владельцу |
| Несанкционированное удаление данных | Потеря истории заказов и абонементов | Средний | Бэкапы, RLS |
| Неправомерный доступ к подписям ЭП | Оспаривание факта заключения договора | Высокий | Хэш неизменяем, хранится отдельно, доступ только Оператору |

**Общий вывод:** обрабатываемые Оператором ПД не относятся к специальным категориям. Наибольший риск представляют геопозиция и адрес доставки. Применяемые технические меры (RLS, HTTPS, rate limiting) соответствуют уровню риска. Обязательна периодическая проверка целостности системы RLS при каждом изменении схемы БД.
