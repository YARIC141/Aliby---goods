# Задания для доработки Aliby Foods

> Файл для ручной корректировки. Каждый раздел можно давать как отдельное задание.

---

## Контекст проекта

- `admin/index.html` — монолитный HTML+JS, ~1360 строк, одна панель администратора
- `client/index.html` — монолитный HTML+JS, ~1234 строки, PWA для клиентов
- Бэкенд — Supabase (REST API + Auth + Edge Functions)
- Сессия хранится в `localStorage` (`aliby_admin_s` / `aliby_client_s`)
- Конфиг Supabase захардкожен в начале каждого файла (URL + ANON_KEY)

## Задание 1 — Экран авторизации для Админки

### Что сейчас есть
В `admin/index.html` строки 703–728 есть функции `doLogin()` и `doLogout()`, но экран авторизации встроен прямо в тело страницы без отдельного дизайна.

### Что нужно сделать
Оформить экран авторизации в **отдельный полноэкранный блок** (`<div id="auth-screen">`), который показывается вместо основного интерфейса, пока пользователь не вошёл.

**Экран должен содержать:**
- Логотип / название «Aliby Admin»
- Поле Email
- Поле Пароль (с кнопкой показать/скрыть)
- Кнопка «Войти»
- Ссылка «Забыли пароль?» → разворачивает форму сброса пароля (поле email + кнопка «Отправить ссылку»)
- Сообщение об ошибке (например, «Нет прав администратора» или «Неверные данные»)

**Логика:**
- При успешном логине проверяется роль (`profiles.role = 'admin'`), если не админ — сессия сбрасывается и показывается ошибка
- Запрос на сброс пароля — `POST /auth/v1/recover` с полем `email`
- При наличии токена в URL (`#access_token=...`) — автоматически показывать форму смены пароля (см. Задание 2)

## Задание 2 — Смена пароля (Admin и Client)

### Что сейчас есть
В `client/index.html` строки 685–692 — функция `doResetPassword()` вызывает `PUT /auth/v1/user` с новым паролем. В `admin/index.html` аналогичной функции нет.

### Что нужно сделать — в Админке

Добавить в шапку (рядом с кнопкой «Выйти») кнопку **«Сменить пароль»**, по клику открывается модальное окно с полями:
- Новый пароль
- Повтор нового пароля
- Кнопки «Сохранить» / «Отмена»

Запрос: `PATCH /auth/v1/user` с телом `{ "password": "..." }` и заголовком `Authorization: Bearer <token>`.

Показывать успех/ошибку внутри модального окна.

**Отдельный случай:** если пользователь пришёл по ссылке из письма (`URL содержит #access_token=...`), показывать форму смены пароля сразу поверх экрана авторизации, а после успешной смены — перейти к обычному логину.

### Что нужно сделать — в Клиенте

В `client/index.html` на экране «Профиль» добавить кнопку **«Сменить пароль»**, которая раскрывает прямо на экране (не в модалке) форму:
- Новый пароль
- Повтор нового пароля
- Кнопки «Сохранить» / «Отмена»

Логика та же — `PATCH /auth/v1/user`. Экран сброса пароля по ссылке из письма уже реализован (строки 685–692), его трогать не нужно.

## Задание 3 — Категории заведений: Администратор

### Что сейчас есть
- Таблица `store_categories` в БД (отдельный справочник типов заведений: кафе, ресторан, бар и т.д.)
- В `admin/index.html` строка 733 — функция `loadStoreCategories()` загружает их для выпадающего списка при создании/редактировании заведения
- Поле `store_category` в таблице `stores` уже существует

### Что нужно сделать

**А) Управление справочником категорий заведений**

Добавить в боковое меню пункт **«Типы заведений»** (или в раздел уже существующих «Категорий»). На экране:
- Таблица с колонками: Название, Иконка (если есть), Действия
- Кнопка «+ Добавить тип»
- Модальное окно: поле Название, поле Иконка (emoji или URL), кнопки «Сохранить» / «Удалить»
- API: `GET /rest/v1/store_categories`, `POST`, `PATCH?id=eq.{id}`, `DELETE?id=eq.{id}`

**Б) Назначение категории заведению**

В модальном окне создания/редактирования заведения (`openStoreModal()`) убедиться, что:
- Есть выпадающий список `<select>` с вариантами из `store_categories`
- Выбранное значение сохраняется в поле `store_category` при `POST`/`PATCH` заведения
- При открытии существующего заведения — выбранная категория подставляется в список

Если список уже есть — проверить и при необходимости исправить баг с загрузкой/сохранением.

## Задание 4 — Категории заведений: Клиент

### Что сейчас есть
В `client/index.html` есть экраны «Карта» и «Заведения». Заведения отображаются в виде списка, но без разделения по типам.

### Что нужно сделать

**А) Фильтрация по типу заведения на экране «Заведения»**

Над списком заведений добавить горизонтальную полосу фильтров (chips / кнопки-теги):
- «Все» — выбрано по умолчанию
- Далее — все типы из `store_categories` (загружаются при старте: `GET /rest/v1/store_categories`)
- При выборе типа — список заведений фильтруется по полю `store_category`
- Фильтрация на клиенте (в уже загруженных данных), без дополнительных запросов к API

**Б) Отображение типа заведения в карточке**

В карточке заведения (на экране «Заведения» и в детальном виде) добавить под названием:
- Иконка + название типа заведения (берётся из загруженного справочника по `store_category` ID)

**В) Фильтрация на карте (опционально)**

Аналогичные chips над картой, при выборе типа — скрывать маркеры не того типа.

## Технические ограничения

- Не использовать сторонние библиотеки (проект работает без сборки и npm)
- Стили писать инлайн или в `<style>` в том же файле
- Всё в двух файлах: `admin/index.html` и `client/index.html`
- API вызовы через функцию `req()` (уже есть в обоих файлах)
- Supabase Auth endpoints: `/auth/v1/token`, `/auth/v1/signup`, `/auth/v1/recover`, `/auth/v1/user`
