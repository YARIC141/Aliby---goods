# Alliby Carry — техническое описание (черновик v1)

Отдельное веб-приложение (со своей native shell, по аналогии с client/admin) для фрилансеров-курьеров. Платформа только помогает продавцу и курьеру найти друг друга и договориться о цене — деньги за доставку между ними проходят полностью вне приложения, Alliby комиссию не берёт и не контролирует.

## 1. Роли

- **Курьер** — фрилансер, открытая самозапись в Alliby Carry (имя, телефон, город), без модерации/верификации при регистрации.
- **Продавец / администратор** — пользователь Alliby Admin, владелец заведения, настраивает способ доставки в редактировании заведения.
- **Покупатель** — клиент Alliby (client app), оформляет заказ с доставкой, видит статус и телефон курьера — без изменений в интерфейсе относительно того, что есть сегодня.
- **Владелец платформы** — модератор по жалобам.

## 2. Профиль курьера

- Город работы (указывается в профиле, определяет, в заказах какого города курьер может участвовать).
- Минимальная ставка вознаграждения — сумма, ниже которой курьер не согласен выполнять заказ. Редактируется в любой момент; изменение применяется только к будущим аукционам, уже идущие не затрагивает.
- Статус: свободен / занят (занят = есть активный незавершённый заказ). Только свободные курьеры участвуют в поиске/аукционе.
- Геопозиция — периодически отправляется на сервер, пока курьер онлайн и свободен (для расчёта расстояния до заведения).

## 3. Настройки доставки заведения (Alliby Admin → редактирование заведения)

Продавец настраивает способ поиска курьера для заказов с доставкой:

**Режим А — «Свой курьер».**
Заведение указывает список ID курьеров (из системы Alliby Carry), которые работают на него постоянно (договорённость о цене — вне приложения). При заказе система назначает заказ тому из списка, кто сейчас свободен; если свободно несколько — ближайшему к заведению. Аукциона и сравнения ставок в этом режиме нет.

**Режим Б — «Найти через Alliby Carry»** (открытый пул курьеров города).
Дополнительный переключатель внутри этого режима:
- **Аукцион** — среди свободных курьеров города, чья минимальная ставка ≤ цены доставки, установленной продавцом для зоны доставки заказа (`delivery_zones.price`), выбирается тот, кто указал наименьшую ставку. Курьер получает **свою** ставку; разница между ставкой и ценой продавца остаётся у продавца — это их дело, приложение в расчётах не участвует.
- **Полная стоимость ближайшему** — среди свободных курьеров, чья минимальная ставка ≤ цены доставки (тот же фильтр по порогу, просто без сравнения ставок между собой), выбирается географически ближайший к заведению. Он получает полную сумму, указанную продавцом.

Также в этом же месте настроек — **радиус поиска** (в каком радиусе от заведения искать курьеров).

## 4. Алгоритм назначения (общий для режима Б)

1. Заказ оформлен и оплачен, доставка активирована.
2. Система ищет среди свободных курьеров нужного города, в радиусе поиска, отфильтрованных по порогу ставки (только для режима Б; в режиме А — только по списку ID заведения).
3. Если подходящих курьеров нет — аукцион/поиск не проводится. Продавцу показывается «Курьеров в зоне не найдено» с возможностью повторить поиск или увеличить радиус.
4. Если подходящие есть — по правилам режима (см. §3) выбирается один курьер, заказ **автоматически назначается** ему (без отдельного шага «принять» — победитель сразу становится исполнителем).
5. Админу и покупателю приходит уведомление, что курьер найден (имя/телефон отображаются так же, как сегодня работает поле `courier_phone`, просто заполняется автоматически, а не вручную).
6. Курьер видит назначенный заказ в Alliby Carry с адресами продавца и покупателя.

## 5. Отказ курьера после назначения

У курьера есть кнопка «Отказаться от заказа» уже после автоматического назначения (телефон сел, не может выполнить и т.д.). При отказе заказ повторно проходит поиск/аукцион среди оставшихся подходящих свободных курьеров (отказавшийся исключается из этой попытки). Если и там никого нет — тот же экран «не найдено» продавцу.

## 6. Статусы (продавец/курьер)

Курьер после назначения сам продвигает свой этап (забрал / доставил) в Alliby Carry — статусы заказа синхронизируются с уже существующими в Alliby Admin/Client (`looking_for_courier` → `on_the_way` → далее как сегодня отмечает продавец при выдаче).

## 7. Модерация

Верификации курьеров при регистрации нет. Продавец может пожаловаться на курьера (жалоба уходит владельцу платформы по почте). Решение о бане курьера принимает владелец платформы вручную, по факту жалобы — автоматических штрафов/рейтинга нет.

## 8. Явно вне рамок v1

- Приложение не участвует в оплате/выплате вознаграждения курьеру — это происходит полностью вне Alliby (наличные/перевод и т.п., по договорённости продавца и курьера).
- Комиссия платформы — отсутствует.
- Верификация личности/статуса самозанятого курьера — не делается платформой.
- Живое отслеживание курьера на карте покупателем — не нужно для v1 (только статус + телефон, как сегодня).
- Один активный заказ на курьера одновременно — без батчинга нескольких заказов в один рейс.
- Отдельная выплатная/платёжная подсистема — не требуется, т.к. деньги через приложение не идут.

## 9. Бренд

- Название: **Alliby Carry**.
- Иконка — в стиле остальных native shell приложений (client/admin), фон бирюзовый, снизу поверх буквы «A» надпись «carry».

## 10. Технические заметки (для проектирования архитектуры, не решения)

- PostGIS уже включён в БД (используется сейчас только для тайлов карты) — есть база для гео-запросов «ближайший курьер», но саму логику (`ST_DWithin`, ближайший сосед) предстоит написать с нуля.
- Push-уведомления (`send-push`) сейчас жёстко заточены под `app='client'` — под Carry нужен третий тип получателя.
- Realtime сейчас реализован через сырой Phoenix websocket по Postgres-изменениям (по одному каналу на пользователя) — для частой геопозиции курьера лучше не писать в таблицу на каждый пинг, а использовать Presence/Broadcast (эфемерные каналы), сохраняя в БД позицию реже (например, при получении заказа/по таймеру).
- Модель сотрудников (`role='employee'`, `employee_store_id`) не подходит для курьера-фрилансера — нужна отдельная сущность аккаунта, привязанная к городу, а не к одному заведению.
