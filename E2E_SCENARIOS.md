# Aliby — сквозные сценарии тестирования (client + admin + carry)

## Идея

`TEST_CASES.md` проверяет API поэндпоинтно. Этот файл — про другое: **реальные
пользовательские сценарии, в которых действие в одном приложении меняет
состояние, видимое в двух других.** Вместо десятков изолированных тест-кейсов —
несколько длинных сценариев, каждый из которых по пути задевает максимум
функций сразу трёх приложений. Цель: минимум сценариев, максимум покрытия за
один прогон.

Обозначения шагов: `[client]` — client/index.html (покупатель), `[admin]` —
admin/index.html (владелец/сотрудник заведения), `[carry]` — carry/index.html
(курьер), `[DB]` — прямая проверка в Supabase Studio/RPC, `[push]` — ожидаемое
push-уведомление.

---

## Действующие лица и тестовые данные (общие для всех сценариев)

| Роль | Кто | Комментарий |
|---|---|---|
| Владелец A | владелец заведения «Тест-кафе» | direction=food, платная подписка платформы активна |
| Сотрудник A | employee, `employee_store_id` = Тест-кафе | логин/пароль, не email |
| Владелец B | владелец «Тест-салон» | direction=services, 2 мастера |
| Мастер B1 | employee с `is_master=true` в Тест-салоне | ведёт услугу «Стрижка», своя цена/расписание |
| Мастер B2 | employee с `is_master=true` в Тест-салоне | ведёт ту же услугу «Стрижка» с другой ценой/расписанием, плюс услугу без мастера (fallback на расписание заведения) |
| Клиент 1 | обычный покупатель | основной сценарист заказов |
| Клиент 2 | второй покупатель | для подарочного абонемента, RLS-проверок |
| Курьер 1 | is_courier=true, город = город Тест-кафе | основной курьер |
| Курьер 2 | is_courier=true, тот же город | для сценария отказа/передиспетчеризации |

Подготовка перед прогоном:
- В Тест-кафе: товар минимум с 2 группами опций (например «Пицца»: размер +
  начинка), `delivery_enabled=true`, зона доставки — один полигон/радиус,
  покрывающий тестовый адрес А, и заведомо НЕ покрывающий адрес Б (вне зоны).
- `delivery_courier_mode` заведения переключается по ходу сценариев (carry /
  manual / own_courier) — см. каждый сценарий.
- Курьер 1 и Курьер 2 прошли онбординг (`register_courier`), подписали оферту.

## Сценарий 1 — Полный цикл заказа с доставкой через Carry (авто-диспетчеризация)

**Что покрывает:** регистрация/вход клиента → опции товара как раздельные
строки корзины → выбор адреса через экран зон доставки → расчёт ETA и
стоимости доставки → онлайн-оплата (T-Bank) → авто-диспетчеризация курьера →
выход курьера на смену и геопинг → приём/доставка заказа курьером → пуши на
каждом статусе → раздельное отображение адреса и опций в деталях заказа у
клиента → гонка `courier_phone`/`looking_for_courier` → статистика заработка
курьера.

1. `[admin]` Владелец A открывает Тест-кафе → Доставка → `delivery_courier_mode = carry`, `delivery_carry_bid_mode = first_found` (или `auction` — см. вариацию ниже), `delivery_search_radius_m` задан с запасом.
2. `[carry]` Курьер 1 включает смену → убедиться, что уходит геопинг (`update_courier_location`, раз в ~90с).
3. `[client]` Клиент 1 логинится, открывает Тест-кафе, добавляет «Пиццу» дважды с **разным выбором опций** (размер S/начинка А, затем размер M/начинка Б).
   - **Проверка:** в корзине — **две отдельные строки**, каждая со своей ценой/qty, а не одна строка с задвоенным количеством.
4. `[client]` Переключает на «Доставка» → строка адреса появляется под кнопками, тапает по ней → открывается экран зон доставки → выбирает адрес А (в зоне).
   - **Проверка:** ETA и стоимость доставки отображаются простым текстом (не в рамке), адрес — читаемый текст, не координаты.
5. `[client]` Оформляет и оплачивает картой (T-Bank sandbox) → заказ создаётся со статусом `paid`.
   - **Проверка:** `is_delivery`, `delivery_address`, `delivery_fee`, `delivery_lat/lng` заполнены корректно в `orders`.
6. `[DB]`/автоматически: триггер авто-диспетчеризации переводит заказ в `looking_for_courier`, запускается `dispatch_order_courier()`.
   - **Проверка:** Курьер 1 (на смене, геопинг свежий, `courier_min_reward <= delivery_fee`, в радиусе) назначается — `carry_courier_id`, `courier_phone`, `carry_courier_reward`, `carry_dispatch_status='assigned'`.
   - `[push]` Курьер 1 получает `carry_order_assigned`.
7. `[admin]` Владелец A видит `looking_for_courier`/назначенного курьера в карточке заказа → физически передаёт заказ курьеру → жмёт «Передано курьеру» → статус `handed_to_courier`.
   - `[push]` Курьер получает `carry_order_handed`.
8. `[carry]` Курьер 1 видит заказ в списке активных → «Принять» → `accepted_by_courier`.
9. `[client]` **Ключевая проверка гонки состояний:** если `courier_phone` уже проставлен, а статус ещё технически `looking_for_courier` (смоделировать через прямой PATCH в БД до шага 8, откатив статус) — клиент должен показывать «Курьер найден» с номером телефона, а не «Ищем курьера».
10. `[carry]` Курьер открывает номер клиента (`get_order_customer_phone` — должен сработать только пока статус в допустимом диапазоне и `carry_courier_id = свой uid`), везёт заказ, жмёт «Доставил» → `issued`, `carry_delivered_at` проставлен.
    - `[push]` Клиент получает `order_issued`.
11. `[client]` В истории заказов и в активном заказе видно: адрес доставки, обе строки товара с их опциями, курьер и его телефон (пока отображался).
12. `[carry]` В статистике/истории курьера заказ учтён: сегодняшний заработок += `carry_courier_reward`, счётчик доставленных +1.

**Вариация внутри того же прогона (не отдельный сценарий):** повторить с `delivery_carry_bid_mode = auction`, минимум 2 подходящих курьера с разным `courier_min_reward` — победить должен курьер с меньшей ставкой, и именно эта сумма должна попасть в `carry_courier_reward`.

## Сценарий 2 — Ручная доставка + свой курьер + абонемент + гейтинг по подписке платформы

**Что покрывает:** создание/редактирование заведения и зон доставки → чекбокс
`delivery_enabled`, заблокированный без активной подписки платформы →
ручное продвижение заказа админом по всем статусам → доставка «своим
курьером» (`own_courier`) без авто-матчинга → адрес вне зоны доставки
(красное предупреждение + заблокированная кнопка «Оплатить») → списание
абонемента при оформлении → просроченный кэш зоны доставки на клиенте →
вход сотрудника с ограничением по своему заведению.

1. `[admin]` Владелец A (или новый тестовый владелец без активной подписки платформы) пытается включить `delivery_enabled` → чекбокс недоступен, пока `platform_subscriptions` не активна.
2. `[admin]` Оформляет/продлевает подписку платформы (`tbank-platform-init`, `type=trial` или `monthly`) → `delivery_enabled` становится доступен, включает; `delivery_courier_mode = own_courier`; добавляет Курьера 2 в список «своих курьеров» заведения (`store_couriers`).
3. `[admin]` Настраивает зону доставки (полигон/радиус) и запоминает/фиксирует цену зоны.
4. `[client]` Клиент 1 покупает абонемент, покрывающий товары Тест-кафе (реальная оплата через `tbank-init-sub`, не «имитация»).
5. `[client]` Открывает Тест-кафе, включает «Доставка», тапает адрес → выбирает адрес Б (вне зоны).
   - **Проверка:** режим доставки остаётся выбранным (не сбрасывается на самовывоз), под строкой адреса — красное «Адрес недоступен для доставки», кнопка «Оплатить» задизейблена.
6. `[client]` Меняет адрес на А (в зоне) → предупреждение исчезает, кнопка активна, к заказу автоматически применяется абонемент (скидка/бесплатные позиции), `payment_method` = `subscription`/`mixed`.
7. `[client]` Оформляет заказ (сумма может быть частично/полностью покрыта абонементом).
8. `[admin]` Владелец A вручную ведёт заказ по статусам: «Взять в работу» (`in_progress`, задаёт время готовности) → «Ищем курьера» (`looking_for_courier`, режим `own_courier` — матчинг идёт только среди `store_couriers`, без city/radius/reward-фильтров) → «Передано курьеру» (`handed_to_courier`).
9. `[carry]` Курьер 2 видит и принимает заказ (`accepted_by_courier`) → доставляет (`issued`).
10. **Проверка устаревания кэша зоны (edge-case, не отдельный сценарий):** пока курьер везёт заказ, Владелец A меняет цену зоны доставки → Клиент 1 в той же открытой сессии (кэш зоны на клиенте — 3 минуты) может какое-то время видеть старую цену для *следующего* заказа; после истечения TTL — новую.
11. `[admin]` Сотрудник A логинится (логин/пароль, не email) → видит только заказы/аналитику/бронирования своего заведения (Тест-кафе), не видит Тест-салон.

## Сценарий 3 — Отказ курьера, повторная диспетчеризация, запись на услугу

**Что покрывает:** отказ курьера от заказа и повторный подбор без
исключённого курьера → «курьеры не найдены» и ручной повтор поиска админом
→ параллельно (для экономии сценариев) — ветка бронирования услуги: мастер,
слот, оплата/перенос, пуши бронирования, отмена.

**Ветка A — курьеры:**
1. `[admin]` Тест-кафе в режиме `carry`; временно понизить пул подходящих курьеров до одного (Курьер 1 на смене).
2. `[client]` Клиент 1 оформляет доставку → авто-диспетч назначает Курьера 1.
3. `[carry]` Курьер 1 жмёт «Отказаться» → заказ возвращается в `looking_for_courier`, попадает в `order_courier_declines`, повторный подбор исключает Курьера 1.
   - Если больше никто не подходит → `carry_dispatch_status='no_couriers_found'`.
4. `[admin]` Владелец A видит кнопку «Повторить поиск курьера» → после того как Курьер 2 вышел на смену, жмёт повтор → на этот раз матчится Курьер 2.
5. `[carry]` Курьер 2 принимает и доставляет как в сценарии 1.

**Ветка B — запись к конкретному мастеру, перенос, отмена, оплата (тот же прогон, другое заведение):**
6. `[admin]` Владелец B настраивает услугу «Стрижка» с привязкой обоих мастеров (`master_services`), у каждого — своя цена и своё недельное расписание (`master_schedules`); вторую услугу оставляет без привязки к мастеру (fallback на общее расписание заведения, `store_schedules`).
7. `[client]` Клиент 1 открывает «Стрижку» → появляется выбор мастера (список из Мастера B1/B2, не общий календарь) → выбирает Мастера B1 → календарь показывает **именно его** свободные слоты и **его** цену услуги.
   - **Проверка:** для того же слота у Мастера B2 доступность/цена — другие (независимые расписания и цены).
   - **Проверка:** попытка подтвердить бронь без выбора мастера — блокируется («Выберите мастера»).
8. `[client]` Бронирует слот у Мастера B1 без немедленной оплаты («забронировать, оплатить позже») → `bookings` создаётся `status='booked'`, `payment_status='unpaid'`, `master_id` = Мастер B1.
   - `[push]` Владелец B/Мастер B1 получает `booking_created`.
9. `[admin]` Владелец B (или сам Мастер B1, т.к. видит только своё заведение) в календаре отмечает бронь оплаченной вручную (`payment_status→paid`) → `[push]` `booking_confirmed`.
10. `[client]` Клиент 1 открывает вторую услугу (без привязанного мастера) → бронирует **с онлайн-оплатой** через `/functions/v1/tbank-init-booking` → сразу `payment_status='paid'` → `[push]` `booking_confirmed` сразу при создании (INSERT с paid).
11. **Перенос («Перенести»)** — проверить оба условия допустимости и оба недопустимых случая:
    - `[client]` Сразу после создания брони у Мастера B1 (в пределах 15 минут от `created_at`) — «Перенести» доступно.
    - `[client]` Отдельно создать бронь на слот, до которого остаётся ≥24ч — «Перенести» доступно и вне 15-минутного окна.
    - `[client]` Создать бронь на слот менее чем через 24ч и подождать >15 минут — «Перенести» должно быть заблокировано с пояснением.
    - При успешном переносе: выбирается новый слот (можно сменить и мастера, напр. с B1 на B2) → **создаётся НОВАЯ запись** в `bookings` с новым `slot_date/slot_start/master_id`, а **старая запись не двигается по времени**, у неё меняется только `status→'rescheduled'`.
    - **Проверка на оплату:** если исходная бронь уже была оплачена (`payment_status='paid'`), перенос **не переносит оплату** на новую запись — новая создаётся с собственным `payment_status` (unpaid/paid по выбранному способу оплаты заново). Явно зафиксировать это как проверяемое поведение (см. блок «известные находки» ниже).
    - **Проверка пуша:** сам факт `status→'rescheduled'` на старой записи push не генерирует — уведомление придёт только по INSERT новой записи (`booking_created`/`booking_confirmed`). Убедиться, что нет дублей/пропущенных пушей.
12. **Отмена — с обеих сторон:**
    - `[client]` Клиент 1 отменяет одну из своих активных броней (`status→'cancelled'`) → `[push]` `booking_cancelled`.
    - `[admin]` Владелец B/Мастер B1 отменяет другую бронь Клиента 1 из своего календаря → тот же переход, тот же `[push]` `booking_cancelled`.
    - **Проверка (известное поведение, не баг теста):** если отменяется уже оплаченная бронь — `payment_status` остаётся `'paid'`, никакого автоматического возврата/refund в системе не происходит (возврат — не через платформу, по договорённости с заведением). Зафиксировать фактическое поведение.
13. `[admin]` Владелец B отмечает одну из непереносимых/неотменённых броней выполненной (`status='completed'`) после визита.

## Сценарий 4 — Аренда, подарочный абонемент, истечение подписки заведения

**Что покрывает:** корзина аренды и чекаут → подарочный абонемент (покупка
одним пользователем, активация другим) → истечение подписки платформы у
владельца в процессе работы → клиентский гейтинг (`applyStoreGate`)
заведения, у которого подписка кончилась, хотя меню всё ещё видно.

1. `[client]` Клиент 1 бронирует товар на аренду (день/час), проверяет доступные слоты, оформляет через `tbank-init-rent`/`tbank-init-rent-cart`.
2. `[admin]` Владелец A/B видит резервацию аренды в своей вкладке, меняет статус (`pending → active → completed`), либо отменяет.
3. `[client]` Клиент 1 покупает подарочный абонемент для Клиента 2 (`gift_status='pending'`).
4. `[client]` Клиент 2 логинится, активирует полученный подарочный абонемент (`activate_gift_subscription`) → `gift_status='activated'`, абонемент доступен к использованию при заказе.
5. `[admin]` Владелец A позволяет подписке платформы истечь (не продлевает, либо тестово откатывает `end_date` в прошлое через Studio) — **не через нормальный UI-флоу отмены**, а именно истечение по дате.
6. `[client]` Клиент 1 заходит в Тест-кафе: заведение и меню всё ещё видны, но корзина/чекаут заблокированы (`is_store_subscription_active` = false → `applyStoreGate`).
7. `[admin]` Владелец A продлевает подписку (`monthly`/`add_store`) → у Клиента 1 при следующем открытии заведения чекаут снова доступен.

## Сценарий 5 — Безопасность: попытки нарушить границы ролей на стыке приложений

**Что покрывает:** RLS/RPC-проверки, специфичные именно для взаимодействия
курьер↔клиент↔админ (в отличие от общих RLS-кейсов в `TEST_CASES.md` UC-11).

1. `[carry]` Курьер, **не назначенный** на заказ, пытается вызвать `get_order_customer_phone`/принять/отказаться/доставить чужой заказ → должен получить отказ (403/RPC-ошибка), а не данные/смену статуса.
2. `[carry]` Курьер пытается вызвать `courier_mark_delivered` на заказе не в статусе `accepted_by_courier`/`on_the_way` (например, ещё `looking_for_courier`) → отказ.
3. `[admin]` Владелец пытается вручную выставить `status='issued'` на заказе, где `carry_courier_id IS NOT NULL` (уже назначен курьер Carry) — должно быть заблокировано RLS-политикой; довести до `issued` можно только курьерским действием.
4. `[admin]` Сотрудник A пытается открыть/изменить заказ или бронирование Тест-салона (чужого `employee_store_id`) — не должен увидеть/суметь изменить.
5. `[client]` Клиент 2 пытается получить телефон курьера или адрес доставки чужого заказа Клиента 1 напрямую через REST (`/rest/v1/orders?id=eq...`) → пустой ответ / отказ.
6. `[DB]` (не через UI, отметить как находку, а не проходной тест-кейс): у курьера выставлен `courier_banned=true` напрямую в БД — убедиться, что забаненный курьер реально исключается из матчинга (`dispatch_order_courier`) и/или не может выйти на смену в `[carry]`, хотя в `admin` сейчас нет интерфейса для простановки бана — зафиксировать как продуктовый пробел, а не баг теста.

## Сценарий 6 — Несколько заведений одного владельца + доп. ингредиенты в товаре

**Что покрывает:** покупка слота на доп. заведение, серверное ограничение
лимита заведений (нельзя обойти напрямую через API), различие между
видимостью конкретного заведения и доступностью корзины у ВСЕХ заведений
владельца, блокировка удаления заведения с активной подпиской, группы опций
товара (одиночный/множественный выбор, обязательность, суммирование цены
доп. ингредиентов), известный пробел валидации обязательной группы.

**Ветка A — второе заведение:**
1. `[admin]` Владелец A (пока имеет только Тест-кафе, `plan_type='platform'` активен) жмёт «Новое заведение» второй раз → до покупки слота — блокировка с предложением оформить подписку доп. заведения (`startPlatformSubscription`/`add_store`).
2. `[admin]` Владелец A оплачивает доп. слот (`tbank-platform-init`, `type='add_store'`) → создаётся `platform_subscriptions` строка `plan_type='store'`, `status` активен, `store_id=NULL`.
3. `[admin]` Создаёт «Тест-кафе-2» → строка слота из шага 2 автоматически линкуется к новому заведению (`store_id` проставляется, FIFO по самому старому свободному слоту).
4. `[DB]` **Проверка обхода лимита:** попытаться создать третье заведение напрямую через `POST /rest/v1/stores` (минуя admin-проверки в UI) без дополнительного оплаченного слота → должно быть отклонено на уровне БД (триггер лимита заведений), а не только клиентской проверкой.
5. `[admin]`/`[DB]` Позволить именно `'store'`-слоту Тест-кафе-2 истечь (не трогая основной `'platform'`-план Владельца A) → **Тест-кафе-2 должно полностью пропасть из поиска/карты клиента** (видимость по заведению), при этом:
   - **Проверка:** Тест-кафе (основное, привязанное к общему платному плану) остаётся видимым и с рабочей корзиной.
   - **Проверка (нетривиальный момент):** гейтинг корзины/чекаута управляется ГЛАВНОЙ `'platform'`-подпиской владельца целиком, а не отдельным слотом заведения — т.е. потенциальный тест «Тест-кафе-2 видимо, но корзина заблокирована» **не проявится именно так**: заведение с истёкшим своим слотом просто исчезает целиком, а не остаётся видимым с заблокированным чекаутом. Зафиксировать фактическое поведение.
6. `[admin]` Владелец A пытается удалить Тест-кафе-2, пока у него ещё активен слот подписки → блокировка «Нельзя удалить — есть активная подписка». Продлить/дождаться корректного состояния и повторить — либо сначала отменить слот, затем удалить.

**Ветка B — доп. ингредиенты (группы опций товара), тот же прогон:**
7. `[admin]` В Тест-кафе-2 создаёт товар «Шаурма» с двумя группами опций: «Размер» — тип `radio` (одиночный выбор), 2-3 варианта, помечена «Обяз.»; «Доп. ингредиенты» — тип `checkbox` (множественный выбор), 3+ варианта (сыр +50₽, бекон +70₽, соус +30₽ и т.п.), тоже помечена «Обяз.».
8. `[client]` Клиент 1 открывает «Шаурму»: группа «Размер» — один вариант уже предвыбран по умолчанию (radio никогда не остаётся пустым); группа «Доп. ингредиенты» — ничего не выбрано изначально, каждый вариант переключается независимо.
9. `[client]` Выбирает несколько доп. ингредиентов сразу (сыр + бекон) → **цена товара должна суммировать `price_add` всех выбранных вариантов** (а не только последнего/первого).
10. `[client]` **Известный пробел, зафиксировать как находку:** попытаться добавить товар в корзину, явно НЕ выбрав ничего в группе «Доп. ингредиенты», помеченной «Обяз.» — по текущей реализации `required` только визуально показывает «*» у названия группы и **не блокирует** добавление в корзину без выбора. Убедиться, действительно ли это происходит (баг или уже исправлено на момент прогона).
11. `[client]` Оформляет заказ с этим товаром → в деталях заказа (клиент) и в модалке заказа (admin) должны быть видны и группа «Размер», и выбранные доп. ингредиенты с их названиями (`order_items.selected_options`).

## Известные находки, требующие внимания при прогоне (не баги теста — зафиксировать результат)

- `buySubscription()` в клиенте — путь с пометкой «Оплата имитирована» (без реального списания T-Bank), отдельно от `checkoutSubs()`/`tbank-init-sub`. Проверить, какой путь реально используется в проде и не всплывает ли демо-кнопка по ошибке в боевом интерфейсе.
- После миграции `20240101000105` в `notify_order_status_push()` для клиента остались пуши только на `in_progress/ready/issued/cancelled` (+ отдельная ветка `handed_to_courier`→курьеру). Явно проверить в сценарии 1/3: приходит ли клиенту пуш на переход в `looking_for_courier`/`on_the_way` — если нет, это может быть осознанным решением, а может — регрессией.
- Отключение смены в `[carry]` — чисто локальный флаг (`localStorage`), не передаётся на сервер. Пока `courier_location_updated_at` свежий (< 5 мин), выключивший смену курьер **может продолжать матчиться**. Явно зафиксировать это поведение как ожидаемое или как баг.
- `profiles.courier_banned` — нет интерфейса в `admin` для простановки. Тестируется только через прямой SQL.
- Группа опций товара с флагом «Обяз.» (`required`) — по коду это **только визуальная пометка** (звёздочка у названия группы); `addToCartFromDetail()` не проверяет, что в такой группе реально что-то выбрано. Клиент может добавить товар без обязательного выбора размера/ингредиента. Зафиксировать как найденный баг, если поведение не изменилось к моменту прогона.
- Перенос записи (`reschedule`) создаёт **новую строку** `bookings`, а не двигает старую — старая просто помечается `status='rescheduled'` с прежними датой/временем. Оплата **не переносится** на новую запись (у неё собственный `payment_status`), и на сам переход `rescheduled` push не отправляется (только на INSERT новой брони). Это осознанная архитектура, но при регрессионном тестировании важно проверять именно так, а не ожидать «сдвига» той же записи.
- Отмена уже оплаченной записи (клиентом или заведением) не запускает никакого возврата/refund в системе — `payment_status` остаётся `'paid'` при `status='cancelled'`. Возврат средств — вне платформы, по договорённости с заведением (это в её условиях использования), не баг.
- Гейтинг `delivery_enabled`/чекаута у **всех** заведений владельца завязан на его ОСНОВНУЮ (`plan_type='platform'`) подписку, а не на отдельный `'store'`-слот конкретного доп. заведения. Слот доп. заведения управляет только **видимостью** этого конкретного заведения (`is_visible`). Из-за этого нельзя получить состояние «заведение видно, но корзина у него отдельно заблокирована из-за истечения именно его слота» — при истечении слота заведение просто пропадает целиком. Стоит явно перепроверить это допущение при прогоне сценария 6, а не считать его данностью только по анализу кода.
