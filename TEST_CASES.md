# Aliby Foods — Тест-кейсы API

> Все тесты выполняются через Swagger UI (`swagger/index.html`).  
> Переменные в формате `{{VAR}}` — подставьте реальные значения из ответов предыдущих шагов.

---

## Условные обозначения

| Значок | Смысл |
|--------|-------|
| `[публичный]` | Только `apikey` заголовок, JWT не нужен |
| `[user]` | Требует Bearer JWT обычного пользователя |
| `[admin]` | Требует Bearer JWT пользователя с ролью `admin` |
| ✅ | Ожидаемый успешный результат |
| ❌ | Ожидаемый отказ (негативный кейс) |

---

## UC-01: Регистрация и аутентификация

**Цель:** убедиться, что регистрация, вход, получение профиля и выход работают.

---

### TC-01-01: Регистрация нового пользователя

**Эндпоинт:** `POST /auth/v1/signup`  
**Доступ:** `[публичный]`

**Тело запроса:**
```json
{
  "email": "testuser@aliby.ru",
  "password": "Test1234"
}
```

**Ожидаемый результат:** ✅ `200 OK`
```json
{
  "access_token": "eyJ...",
  "refresh_token": "...",
  "token_type": "bearer",
  "expires_in": 3600,
  "user": {
    "id": "{{USER_ID}}",
    "email": "testuser@aliby.ru"
  }
}
```

**Сохранить:** `access_token` → вставить в Swagger Authorize → `bearerAuth`

---

### TC-01-02: Регистрация с уже существующим email

**Эндпоинт:** `POST /auth/v1/signup`  
**Доступ:** `[публичный]`

**Тело:**
```json
{
  "email": "testuser@aliby.ru",
  "password": "AnotherPass1"
}
```

**Ожидаемый результат:** ❌ `422 Unprocessable Entity`  
Тело ответа содержит сообщение об ошибке (email уже используется).

---

### TC-01-03: Регистрация со слабым паролем (< 6 символов)

**Эндпоинт:** `POST /auth/v1/signup`

**Тело:**
```json
{
  "email": "weak@aliby.ru",
  "password": "123"
}
```

**Ожидаемый результат:** ❌ `422` — пароль не соответствует минимальной длине.

---

### TC-01-04: Вход по email и паролю

**Эндпоинт:** `POST /auth/v1/token?grant_type=password`  
**Доступ:** `[публичный]`

**Параметр запроса:** `grant_type = password`

**Тело:**
```json
{
  "email": "testuser@aliby.ru",
  "password": "Test1234"
}
```

**Ожидаемый результат:** ✅ `200 OK` с новым `access_token`.

**Сохранить:** `access_token`, `refresh_token`.

---

### TC-01-05: Вход с неверным паролем

**Эндпоинт:** `POST /auth/v1/token?grant_type=password`

**Тело:**
```json
{
  "email": "testuser@aliby.ru",
  "password": "WrongPass"
}
```

**Ожидаемый результат:** ❌ `400 Bad Request` — неверные учётные данные.

---

### TC-01-06: Обновление токена через refresh_token

**Эндпоинт:** `POST /auth/v1/token?grant_type=refresh_token`

**Тело:**
```json
{
  "refresh_token": "{{REFRESH_TOKEN}}"
}
```

**Ожидаемый результат:** ✅ `200 OK` — новая пара `access_token` / `refresh_token`.

---

### TC-01-07: Получение данных текущего пользователя

**Эндпоинт:** `GET /auth/v1/user`  
**Доступ:** `[user]` — необходим Bearer JWT

**Ожидаемый результат:** ✅ `200 OK`
```json
{
  "id": "{{USER_ID}}",
  "email": "testuser@aliby.ru",
  "role": "authenticated"
}
```

---

### TC-01-08: Запрос на сброс пароля

**Эндпоинт:** `POST /auth/v1/recover`

**Тело:**
```json
{
  "email": "testuser@aliby.ru"
}
```

**Ожидаемый результат:** ✅ `200 OK` — письмо отправлено (в Supabase local → InBucket).

---

### TC-01-09: Выход из системы

**Эндпоинт:** `POST /auth/v1/logout`  
**Доступ:** `[user]`

**Ожидаемый результат:** ✅ `204 No Content`

## UC-02: Просмотр заведений (карта)

**Цель:** публичный доступ к списку кафе без авторизации.

---

### TC-02-01: Получить все заведения

**Эндпоинт:** `GET /rest/v1/stores`  
**Доступ:** `[публичный]`  
**Параметры:** `select=*`, `order=name.asc`

**Ожидаемый результат:** ✅ `200 OK` — массив объектов `Store`, каждый содержит `id`, `name`, `address`, `latitude`, `longitude`.

---

### TC-02-02: Поиск заведения по названию

**Эндпоинт:** `GET /rest/v1/stores`  
**Параметры:** `name=ilike.*кофе*`

**Ожидаемый результат:** ✅ `200 OK` — только заведения, в названии которых есть «кофе» (без учёта регистра).

---

### TC-02-03: Получить конкретное заведение по ID

**Эндпоинт:** `GET /rest/v1/stores`  
**Параметры:** `id=eq.{{STORE_ID}}`, `select=*`

**Ожидаемый результат:** ✅ `200 OK` — массив из одного элемента.

---

### TC-02-04: Пагинация списка заведений

**Эндпоинт:** `GET /rest/v1/stores`  
**Параметры:** `limit=2`, `offset=0`, `order=name.asc`

Затем: `limit=2`, `offset=2`

**Ожидаемый результат:** ✅ каждый запрос возвращает не более 2 заведений, вторая страница — следующие 2.

## UC-08: Пользователь — заказ с использованием абонемента

### TC-08-01: Получить активные абонементы пользователя

**Доступ:** `[user]`  
**Параметры:** `user_id=eq.{{USER_ID}}`, `status=eq.active`, `select=*,subscriptions(*)`

**Ожидаемый результат:** ✅ `200 OK` — активные абонементы с деталями плана.

---

### TC-08-02: Проверить применимость абонемента к заказу

**Эндпоинт:** `POST /functions/v1/validate-subscription`  
**Доступ:** `[user]`

**Тело:**
```json
{
  "user_subscription_id": "{{USER_SUBSCRIPTION_ID}}",
  "order_items": [
    {
      "menu_item_id": "{{MENU_ITEM_ID}}",
      "quantity": 1
    }
  ]
}
```

**Ожидаемый результат (применим):** ✅ `200 OK`
```json
{
  "valid": true,
  "discount_amount": 250.00
}
```

**Ожидаемый результат (временное ограничение):** ✅ `200 OK`
```json
{
  "valid": false,
  "reason": "Действует только с 08:00 до 20:00"
}
```

---

### TC-08-03: Создать заказ с абонементом

**Эндпоинт:** `POST /rest/v1/orders`  
**Заголовок:** `Prefer: return=representation`

**Тело:**
```json
{
  "user_id": "{{USER_ID}}",
  "store_id": "{{STORE_ID}}",
  "total_amount": 0,
  "payment_method": "subscription",
  "applied_user_subscription_id": "{{USER_SUBSCRIPTION_ID}}"
}
```

**Ожидаемый результат:** ✅ `201 Created` — заказ со статусом `pending`.  
**Сохранить:** `id` → `{{ORDER_ID_SUB}}`

---

### TC-08-04: Списать использование абонемента

**Эндпоинт:** `POST /functions/v1/redeem-subscription`  
**Доступ:** `[user]`

**Тело:**
```json
{
  "user_subscription_id": "{{USER_SUBSCRIPTION_ID}}",
  "order_id": "{{ORDER_ID_SUB}}",
  "amount_discounted": 250.00
}
```

**Ожидаемый результат:** ✅ `200 OK`
```json
{
  "redemption_id": "{{REDEMPTION_ID}}",
  "remaining_uses": 4
}
```

**Проверка:** статус заказа должен смениться на `paid` — `GET /rest/v1/orders?id=eq.{{ORDER_ID_SUB}}`.

---

### TC-08-05: Повторное списание для одного заказа

**Выполнить TC-08-04 ещё раз с тем же `order_id`.**

**Ожидаемый результат:** ❌ `409 Conflict` — абонемент уже применён к этому заказу.

---

### TC-08-06: Использование исчерпанного абонемента

**Предусловие:** `remaining_uses = 0`.

**Тело:** аналогично TC-08-04.

**Ожидаемый результат:** ❌ `409 Conflict` — лимит использований исчерпан.

---

### TC-08-07: История списаний по абонементу

**Эндпоинт:** `GET /rest/v1/subscription_redemptions`  
**Параметры:** `user_subscription_id=eq.{{USER_SUBSCRIPTION_ID}}`, `order=redeemed_at.desc`

**Ожидаемый результат:** ✅ `200 OK` — записи о списаниях.

## UC-11: Безопасность — RLS и авторизация (негативные кейсы)

**Цель:** убедиться, что RLS политики работают корректно.

---

### TC-11-01: Создать заведение без авторизации

**Эндпоинт:** `POST /rest/v1/stores`  
**Убрать Bearer токен из Swagger Authorize.**

**Тело:**
```json
{
  "name": "Хакерское кафе",
  "address": "ул. Взлома, 0",
  "latitude": 0,
  "longitude": 0
}
```

**Ожидаемый результат:** ❌ `401` или `403`.

---

### TC-11-02: Обычный пользователь создаёт заведение

**Использовать JWT обычного пользователя (не admin).**

**Ожидаемый результат:** ❌ `403` — RLS разрешает INSERT только администраторам.

**Тело:**
```json
{
  "status": "issued"
}
```

**Ожидаемый результат:** ❌ `403` или `204` с нулевыми изменениями (RLS не пропустит).

---

### TC-11-04: Анонимный пользователь читает user_subscriptions

**Эндпоинт:** `GET /rest/v1/user_subscriptions`  
**Без Bearer JWT.**

**Ожидаемый результат:** ❌ `401` — таблица требует авторизации.

---

### TC-11-05: Пользователь списывает чужой абонемент

**POST /functions/v1/redeem-subscription** с `user_subscription_id`, принадлежащим другому пользователю.

**Ожидаемый результат:** ❌ `403` или `409` — функция проверяет владельца.
