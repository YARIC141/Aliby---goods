Акторы
•	Неавторизованный пользователь – любой посетитель приложения.
•	Авторизованный пользователь – зарегистрировавшийся и вошедший клиент.
•	Администратор – управляет заведениями, меню, абонементами и заказами.

Use Cases для Неавторизованного пользователя
UC1 – Просмотр карты с заведениями
Описание: Пользователь открывает приложение и видит Яндекс.Карту с маркерами заведений.
Основной поток:
1.	Загрузка списка заведений из Supabase.
2.	Отображение маркеров.
3.	Управление картой (перемещение, масштаб).
4.	При клике на маркер – балун с названием и кнопкой «Подробнее».
UC1.1 – Просмотр списка заведений поблизости + поиск заведений по названию
Описание: Пользователь может увидеть список заведений, отсортированных по расстоянию от его текущего местоположения, а также найти заведение по названию через строку поиска.
Основной поток:
1.	Пользователь предоставляет разрешение на геолокацию (если не дано – отображается список без сортировки по расстоянию или с предложением ввести адрес вручную).
2.	Приложение определяет координаты пользователя и загружает список всех заведений с их координатами.
3.	Вычисляется расстояние от пользователя до каждого заведения (по прямой или с учётом дорог – опционально).
4.	Заведения отображаются в виде списка (или переключатель «Список / Карта»):
o	Название, адрес, примерное расстояние (например, «450 м»).
o	Возможность сортировки: «по расстоянию» (по умолчанию), «по названию», «по рейтингу» (если есть).
5.	Вверху списка находится поле поиска по названию заведения (фильтрация в реальном времени).
6.	При клике на элемент списка – переход к странице заведения (UC2).
Альтернативные потоки:
•	Если пользователь отказался от геолокации – отображается список всех заведений без сортировки по расстоянию, с предложением ввести адрес вручную (или кнопка «Включить геолокацию»).
•	Если поиск не дал результатов – отображается сообщение «Заведения не найдены».
UC2 – Просмотр информации о заведении
Описание: Детальная карточка заведения (часы работы, телефон, адрес, соцсети).
Основной поток:
1.	Клик по маркеру или выбор из списка (UC1 или UC1.1).
2.	Отображение: название, адрес, телефон, часы работы, ссылки на соцсети.
3.	Кнопки «Меню» и «Абонементы» (доступны для просмотра всем).
UC3 – Просмотр меню заведения (с поиском и фильтрацией)
Описание:
Пользователь (авторизованный или нет) просматривает список блюд/напитков заведения с возможностью поиска по названию, фильтрации по категориям, цене и другим параметрам.
Основной поток:
1.	Пользователь на странице заведения нажимает «Меню».
2.	Загружаются все позиции меню для этого заведения (каждая позиция содержит: фото, название, цену, краткое описание, категорию).
3.	Над списком отображаются элементы управления:
o	Поле поиска (по названию блюда, в реальном времени).
o	Фильтр по категориям (выпадающий список или чипсы: «Напитки», «Десерты», «Горячее», «Салаты» и т.д. – категории, заданные администратором для данного заведение).
o	Фильтр по цене (слайдер или два поля: от – до).
o	(Опционально) Сортировка (по цене: сначала дешёвые/дорогие, по названию).
4.	При вводе текста в поиск или изменении фильтров список блюд динамически обновляется (без перезагрузки страницы).
5.	Пользователь может сбросить все фильтры кнопкой «Сбросить».
6.	При клике на карточку блюда – переход к деталям блюда (UC4).
7.	Для неавторизованного пользователя кнопка «Заказать» неактивна или заменяется предложением войти.
Альтернативные потоки:
•	Если по заданным критериям ничего не найдено – отображается сообщение «Ничего не найдено. Попробуйте изменить фильтры».
•	Если у заведения нет блюд – показывается надпись «Меню пока не добавлено».
UC4 – Просмотр деталей блюда
Описание: Полная информация о блюде.
Основной поток:
1.	Выбор блюда из меню.
2.	Экран с фото, названием, полным описанием, ценой, составом.
3.	Для авторизованных – кнопка «Добавить в корзину»; для неавторизованных – «Войдите, чтобы заказать».
UC5 – Просмотр абонементов заведения (публичный)
Описание: Просмотр доступных абонементов (типы, условия, цена).
Основной поток:
1.	Нажатие «Абонементы» на странице заведение.
2.	Отображение карточек: название, цена, срок действия, основные условия.
3.	При клике – детали абонемента (UC6).
4.	Кнопка «Купить» ведёт на страницу входа.
UC6 – Просмотр деталей и условий абонемента
Описание: Детальное описание абонемента.
Основной поток:
1.	Выбор абонемента из списка.
2.	Показывается полная информация: что даёт, как использовать, срок действия, цена, ограничения (дни, время, покрываемые блюда).
3.	Кнопка «Приобрести» (только для авторизованных).

Use Cases для Авторизованного пользователя
UC7 – Регистрация / авторизация
Описание: Создание учётной записи и вход.
Основной поток:
1.	Нажатие «Войти» (или при попытке заказа/покупки).
2.	Выбор способа: email + пароль (или телефон – упрощённо email).
3.	Ввод данных, проверка, создание сессии.
4.	Перенаправление на предыдущий экран.
Альтернативный поток – восстановление пароля:
•	«Забыли пароль?» → ввод email → Supabase Auth отправляет письмо → переход по ссылке → новый пароль.
UC8 – Просмотр истории заказов
Описание: Список ранее сделанных заказов блюд.
Основной поток:
1.	Личный кабинет → «История заказов».
2.	Отображение заказов: дата, заведение, список блюд, сумма, статус.
3.	Возможность перейти к деталям заказа.
UC9 – Просмотр приобретённых абонементов
Описание: Список купленных абонементов с остатками использований (с учётом дневных лимитов и общих правил).
Основной поток:
1.	Раздел «Мои абонементы».
2.	Для каждого абонемента: название заведения, тип, дата покупки, срок действия, остаток использований (например, «3 из 5 кофе осталось»), также отображается информация о возможных временных ограничениях и покрываемых блюдах.
3.	При клике – детали использования и условия.
UC10 – Добавление блюд в корзину и оформление заказа (самовывоз)
Описание: Заказ готовых блюд из меню.
Основной поток:
1.	Просмотр меню заведения, добавление блюд в корзину с указанием количества.
2.	В корзине: список, сумма, возможность изменения/удаления.
3.	Подтверждение заказа: указание заведения (фиксированное) и времени самовывоза (опционально).
4.	Переход к оплате (UC12) или применение абонемента (UC13).
5.	После оплаты/списания абонемента заказ сохраняется в истории.
Постусловие: статус заказа «Оплачен, ожидает приготовления» (или «Подтверждён по абонементу»).
UC11 – Покупка абонемента
Описание: Приобретение абонемента заведения.
Основной поток:
1.	Выбор абонемента, нажатие «Купить».
2.	Переход к оплате (UC12).
3.	После успешной оплаты абонемент активируется (дата начала – момент покупки или позже по выбору).
4.	Абонемент появляется в разделе «Мои абонементы» с корректными остатками и сроком действия.
UC12 – Оплата заказа или абонемента (детальный)
Основной поток:
1.	Пользователь инициирует оплату:
o	Для заказа: в корзине нажимает «Оплатить» (если не применён абонемент полностью).
o	Для абонемента: на странице абонемента нажимает «Купить».
2.	Клиентское PWA отправляет запрос в Edge Function create-payment с типом (order или subscription) и ID сущности.
3.	Edge Function проверяет существование и сумму:
o	Для заказа: запись в orders со статусом pending, сумма = total_amount.
o	Для абонемента: создаётся запись в user_subscriptions со статусом pending; если абонемент платный, цена берётся из subscriptions.price.
4.	Edge Function обращается к API платёжного шлюза (ЮKassa), передавая сумму, описание, return_url для редиректа.
5.	Шлюз возвращает payment_id и confirmation_url.
6.	Edge Function сохраняет в таблицу payments запись с status = 'pending', связывая её с order_id или user_subscription_id, и возвращает клиенту confirmation_url.
7.	Клиент открывает платёжную страницу шлюза (redirect или WebView).
8.	Пользователь вводит платёжные данные на стороне шлюза.
9.	После завершения шлюз:
o	перенаправляет браузер на return_url (с параметрами, например ?order_id=...),
o	отправляет webhook (событие payment.succeeded) на URL нашей Edge Function payment-webhook.
10.	Edge Function payment-webhook:
o	Проверяет подлинность уведомления.
o	Находит запись в payments по provider_transaction_id или метаданным, обновляет её статус на succeeded.
o	Если платёж был за заказ: меняет статус заказа на paid, фиксирует время.
o	Если платёж был за абонемент:
	Генерирует уникальный код (например, SUB- + 8 случайных символов) и сохраняет в user_subscriptions.code.
	Меняет статус user_subscriptions на active.
	Заполняет start_date и end_date на основе duration_days или текущей даты.
	Устанавливает remaining_uses = total_uses.
11.	Клиент, вернувшись на return_url, отображает экран ожидания и подписывается на изменения своей записи через Supabase Realtime (либо опрашивает статус). Когда статус меняется на paid/active, показывает подтверждение.
12.	Для заказа – пользователь видит заказ в истории с пометкой «Оплачен». Для абонемента – в разделе «Мои абонементы» появляется карточка с кодом, который можно предъявить в заведении.
Альтернативные потоки:
•	Платёж отклонён / отменён – шлюз отправляет webhook payment.canceled, статус заказа/абонемента возвращается к исходному (удаляется/помечается как cancelled).
•	Webhook не доставлен – реализуется фоллбэк: периодическая сверка статуса платежа по API шлюза для записей в payments, ожидающих более N минут.
UC13 – Использование абонемента при заказе (списание) –
Описание: Пользователь при оформлении заказа может применить подходящий абонемент. Система проверяет покрытие блюд, временные ограничения, лимиты (дневные, интервальные, минимальную сумму заказа).
Основной поток:
1.	В корзине перед оплатой предлагается «Применить абонемент».
2.	Отображаются все активные абонементы пользователя для данного заведение, которые хотя бы частично подходят под состав корзины.
3.	Пользователь выбирает абонемент.
4.	Система выполняет комплексную проверку:
o	Покрытие блюд: каждое блюдо должно входить в условия абонемента (все блюда / конкретные блюда / категории / теги). Если есть исключения – проверка не проходит.
o	Временные ограничения: текущий день недели, время суток – соответствуют заданным.
o	Лимиты: остаток использований > 0, не превышен дневной лимит, соблюдён минимальный интервал с предыдущим списанием, сумма заказа ≥ минимальной (если задано).
5.	Если проверки пройдены:
o	Сумма заказа обнуляется (или уменьшается – для сценария частичного покрытия, опционально).
o	Уменьшается remaining_uses и used_today, записывается факт списания.
o	Заказ оформляется без дополнительной оплаты.
6.	Если проверка не пройдена – показывается конкретная причина («Абонемент не покрывает блюдо X», «Действует только по будням», «Дневной лимит исчерпан» и т.д.). Пользователь может выбрать другой абонемент или перейти к обычной оплате.
Альтернативный поток (частичное покрытие): Если абонемент покрывает только часть блюд, то покрытые становятся бесплатными, остальные оплачиваются через UC12.
UC14 – Просмотр контактов и способов связи с заведением
Описание: Пользователь может посмотреть телефон, адрес, ссылки на соцсети заведения.
Основной поток:
1.	На странице заведение раздел «Контакты».
2.	Доступен номер телефона (кликабельный), адрес (построение маршрута), иконки соцсетей.

Use Cases для Администратора
UC15 – Управление заведениями (CRUD)
Описание: Добавление, редактирование, удаление заведений.
Основной поток:
1.	Вход в админ-панель.
2.	Создание нового заведения: название, адрес, координаты, телефон, часы работы, соцсети.
3.	Сохранение – данные появляются в БД и на карте.
UC15.1 – Выбор заведения администратором из списка
Описание:
Администратор перед выполнением операций с конкретным заведением (управление меню, абонементами, заказами) выбирает нужное заведение из списка доступных. Это действие является предварительным для многих других Use Cases администратора (UC16, UC17, UC18, UC20).
Актор: Администратор
Предусловия:
•	Администратор авторизован в админ-панели.
•	В системе создано хотя бы одно заведение (через UC15).
Основной поток:
1.	Администратор входит в админ-панель и видит главный экран управления.
2.	В левом меню или в верхней панели находится выпадающий список «Выбрать заведение» (или виджет со списком всех заведений).
3.	По умолчанию может быть выбрано «Все заведение» (для некоторых разделов, например, общих отчётов) или последнее использованное заведение.
4.	Администратор раскрывает список и видит все заведения, отсортированные по названию (или по дате создания). Каждый элемент списка содержит название, адрес (кратко).
5.	Администратор кликает по нужному заведению.
6.	Система запоминает выбор и переключает контекст:
o	Во всех последующих действиях (управление меню, абонементами, просмотр заказов этого заведения) данные фильтруются по выбранному заведение.
o	Интерфейс может отображать название текущего заведения в шапке (например, «Работаем с: Кофейня “Уголок”»).
7.	Администратор может в любой момент сменить заведение, повторив шаги 4–6. Выбор сохраняется до смены или до выхода из админ-панели.
Альтернативные потоки:
•	Нет заведений в системе – список пуст, отображается сообщение «Сначала добавьте заведение (UC15)». Кнопки управления меню/абонементами недоступны или ведут к UC15.
•	Поиск по списку – если заведений много, в выпадающем списке есть поле поиска по названию для быстрого нахождения.
•	Недавние заведение – можно показать 3–5 последних использованных заведений в отдельной группе.
•	Переход по прямой ссылке – администратор может открыть админ-панель с параметром ?store_id=..., тогда выбор происходит автоматически без ручного выбора.
Постусловия:
Выбранное заведение становится контекстом для всех операций CRUD в разделах «Меню», «Абонементы», «Заказы», «Статистика использования абонементов».
Связь с другими Use Cases:
•	UC15 (Управление заведениями) пополняет список.
•	UC16 (Управление меню) требует предварительного выбора заведения.
•	UC17 (Управление абонементами) требует предварительного выбора заведения.
•	UC18 (Просмотр заказов) фильтрует заказы по выбранному заведению.
•	UC20 (Статистика абонементов) может отображать данные по выбранному заведению.
Примечание по реализации:
•	Хранить выбранное заведение в состоянии приложения (например, в контексте React/Vue) или в URL-параметрах.
•	Если администратор не выбрал заведение, а пытается перейти к UC16 – перенаправлять на выбор заведения или показывать модальное окно с выбором.

UC16 – Управление меню (CRUD) с поддержкой категорий
Описание:
Администратор добавляет, редактирует, удаляет позиции меню для каждого заведения. При этом он назначает блюду категорию из списка, созданного администратором (глобально или для конкретного заведение). Также администратор может управлять самими категориями.
Основной поток:
1.	Администратор входит в админ-панель и выбирает раздел «Управление меню».
2.	Выбирает конкретное заведение из списка (или создаёт новое заведение предварительно через UC15).
3.	Для управления категориями (если нужно добавить новую категорию) переходит в подраздел «Категории» (см. UC16.1).
4.	Для добавления/редактирования блюда:
o	Заполняет поля:
	Название (текст, обязательное)
	Описание (текст)
	Цена (число)
	Категория (выпадающий список из существующих категорий для данного заведения или общих). Поле обязательное.
	Фото (загружает в Supabase Storage)
	Доступность (флаг «В наличии» – по умолчанию true)
o	Сохраняет блюдо.
5.	При редактировании – можно изменить любые поля, включая категорию.
6.	При удалении блюда – подтверждение, затем блюдо скрывается из публичного меню (или физически удаляется).
Альтернативные потоки:
•	Если категория ещё не создана – администратор может перейти к созданию категории (через UC16.1) прямо из выпадающего списка (кнопка «+ Новая категория»).
•	При удалении категории, к которой привязаны блюда, система предлагает либо переназначить блюда на другую категорию, либо запретить удаление (в зависимости от политики).

UC16.1 – Управление категориями меню 
Описание:
Администратор создаёт, редактирует и удаляет категории для блюд. Категории могут быть глобальными (для всех заведение) или привязанными к конкретному заведение.
Основной поток:
1.	Администратор в админ-панели переходит в раздел «Категории меню».
2.	Видит список существующих категорий (для выбранного заведения или глобальные).
3.	Нажимает «Создать категорию»:
o	Вводит название (например, «Напитки», «Десерты»).
o	Выбирает тип: «Общая» (доступна всем заведениям) или «Для конкретного заведение».
4.	Сохраняет – категория появляется в списке доступных при создании/редактировании блюд.
5.	Может редактировать название или удалить категорию (с проверкой, есть ли блюда с этой категорией – при удалении требуется подтверждение и переназначение блюд).

Обновлённая схема данных (дополнения к таблице menu)
В таблицу menu (или dishes) добавить поле:
•	category_id UUID (ссылка на таблицу categories)
Таблица categories:
•	id UUID PK
•	name TEXT
•	store_id UUID NULL (если NULL – глобальная категория, если указан – категория только для этого заведения)
•	created_at TIMESTAMP
Индексы для поиска и фильтрации: по category_id, по price, полнотекстовый поиск по name.
Обновлённый UC17 – Управление абонементами (CRUD) с гибкой настройкой и тогглом для расширенных опций
Описание:
Администратор создаёт и настраивает абонементы. Для упрощения интерфейса основные поля (название, цена, срок действия, количество использований) отображаются сразу, а более сложные условия (покрытие блюд, временные ограничения, лимиты) спрятаны за тогглом «Показать расширенные настройки». Это не пугает администратора множеством параметров при первом знакомстве.
Основной поток:
1.	Администратор входит в админ-панель → раздел «Абонементы → Управление».
2.	Нажимает «Создать абонемент» или выбирает существующий для редактирования.
3.	Базовые поля (видны всегда):
o	Название (обязательное).
o	Цена (рубли).
o	Срок действия: либо количество дней с момента покупки, либо фиксированная дата окончания.
o	Общее количество использований (0 – безлимит).
4.	Ниже размещён тоггл (чекбокс/переключатель) «Расширенные настройки» (по умолчанию выключен).
o	Если тоггл выключен – администратор видит только базовые поля и может сохранить абонемент с упрощёнными правилами: абонемент действует на все блюда в заведение, без временных и дневных лимитов. Это подходит для простых случаев (например, «кофе-карта на 5 чашек»).
5.	Если администратор включает тоггл, под ним раскрываются блоки с расширенными настройками:
o	Условия покрытия (какие блюда/категории/теги входят или исключаются).
o	Временные ограничения (дни недели, интервалы, запрещённые даты).
o	Лимиты использования (максимум списаний в день, минимальный интервал между списаниями, минимальная сумма заказа для применения).
o	(Опционально) Дополнительные правила (например, «не комбинировать с другими скидками»).
6.	Администратор заполняет нужные расширенные поля. Каждый блок можно свёрнуть/развернуть внутри расширенных настроек для удобства.
7.	После заполнения всех полей администратор сохраняет абонемент. Система проверяет корректность (например, если выбрано покрытие «Конкретные блюда», должны быть отмечены хотя бы одно блюдо).
8.	Абонемент становится доступен для покупки пользователям, и при попытке использования проверяются все заданные условия (включая расширенные).
Альтернативные потоки:
•	Если администратор сохраняет абонемент с выключенным тогглом, то в БД автоматически записываются условия по умолчанию:
o	покрытие = «все блюда»;
o	временные ограничения = отсутствуют;
o	лимиты = отсутствуют.
•	При редактировании ранее созданного абонемента, если в нём были расширенные настройки, тоггл при загрузке формы включается автоматически, и соответствующие блоки отображаются заполненными.
•	Если администратор выключает тоггл при редактировании абонемента, который уже содержал расширенные правила, система предупреждает: «При выключении расширенных настроек все дополнительные условия (покрытие, временные лимиты и т.д.) будут сброшены на значения по умолчанию. Продолжить?» При подтверждении – правила сбрасываются.
UC18 – Просмотр и управление заказами
Описание: Администратор видит список заказов, меняет их статус.
Основной поток:
1.	Админ-панель → «Заказы».
2.	Список новых заказов: номер, заведение, состав, сумма, статус (оплачен / по абонементу).
3.	Возможность перевести в статус «Готов», «Выдан».
4.	После выдачи заказ уходит в историю.
UC19 – Просмотр покупок абонементов (отчёт)
Описание: Администратор видит, кто и когда купил абонементы.
Основной поток:
1.	Раздел «Абонементы → Продажи».
2.	Список транзакций: пользователь, заведение, дата, цена, статус активации.
UC20 – Просмотр статистики использования абонементов – новый
Описание: Администратор анализирует востребованность абонементов по данным о списаниях.
Основной поток:
1.	Раздел «Абонементы → Статистика».
2.	Выбор заведения (или всех) и периода.
3.	Отображение:
o	Количество проданных абонементов каждого типа;
o	Общая выручка;
o	Количество использований (списаний);
o	Среднее время активации (от покупки до первого использования);
o	Доля абонементов, не использованных полностью.
4.	При клике на абонемент – детализация по пользователям (количество списаний, даты).
5.	Экспорт отчёта в CSV/Excel.

Примечания по реализации (обновлённые)
•	Абонементы – базовые поля в таблице subscriptions.
•	Гибкие условия хранятся в связанных таблицах:
o	subscription_coverages – покрытие блюд/категорий/тегов и исключения.
o	subscription_time_rules – временные ограничения.
o	subscription_limits – дневные лимиты, интервалы, минимальная сумма заказа.
•	Логика проверки реализована в серверной функции (Edge Function Supabase или PL/pgSQL).
•	Таблицы покупок и списаний: user_subscriptions (с полями used_today, last_used_at) и subscription_redemptions.
•	Заказы – таблица orders (без доставки, самовывоз).
•	Оплата – интеграция с ЮKassa/Stripe через Edge Functions.
•	Восстановление пароля – через Supabase Auth.
