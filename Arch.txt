Итоговое описание системы: PWA-приложение для отображения заведений на карте с админ-панелью
1. Цель и назначение системы
Разработать бесплатное для запуска и эксплуатации приложение, состоящее из двух прогрессивных веб-приложений (PWA):
•	Клиентское PWA – для конечных пользователей, позволяет просматривать заведения на Яндекс.Карте, знакомиться с их меню (фото, описание, цена).
•	Административное PWA – для управляющего, позволяет добавлять/редактировать/удалять заведения и позиции меню, загружать фотографии.
Оба приложения общаются с единым бэкендом через API. Система не требует платы за хостинг или сервер на начальном этапе, работает офлайн в ограниченном режиме и использует открытые или условно-бесплатные сервисы.

2. Общая архитектура (текстовое описание)
Система построена по принципу «клиент-сервер» с выделенной backend-платформой и статическим хостингом фронтендов.
Ключевые элементы:
•	Клиентское PWA – React-приложение, работающее в браузере, загруженное с GitHub Pages. Отображает Яндекс.Карту с маркерами заведений (координаты из БД). Поддерживает офлайн-кеширование статики и данных благодаря Service Worker.
•	Админское PWA – аналогичное React-приложение, но с формами управления и загрузкой фото. Также хостится на GitHub Pages, но на отдельном поддомене или в отдельном репозитории (или с разграничением по ролям внутри одного приложения).
•	Бэкенд – предоставляется сервисом Supabase (Backend-as-a-Service). Включает:
o	PostgreSQL для хранения заведений, позиций меню, профилей пользователей.
o	Автоматически генерируемый REST API (PostgREST) для CRUD-операций.
o	Supabase Auth для регистрации/входа и управления ролями (user / admin).
o	Supabase Storage для хранения фотографий меню.
o	Edge Functions (опционально) для геокодирования (адрес → координаты) через Яндекс.Геокодер.
•	Внешние сервисы:
o	Яндекс.Карты JS API – отображение карты на клиенте, геолокация.
o	GitHub Pages – хостинг статических файлов обоих PWA.
o	GitHub Actions – CI/CD для автоматической сборки и деплоя при пуше в репозиторий.
Взаимодействие компонентов:
1.	Пользователь заходит на https://username.github.io/client-pwa/. GitHub Pages отдаёт index.html, загружается React, инициализируется Service Worker.
2.	Клиентское PWA через Supabase JS-клиент аутентифицируется (при необходимости) и запрашивает список заведений (/rest/v1/stores). Supabase возвращает JSON с названиями, координатами, адресами.
3.	Приложение подключает Яндекс.Карты (API-ключ, ограниченный доменом), наносит маркеры.
4.	При клике на маркер – загружает меню (/rest/v1/menu_items?store_id=eq.xxx), отображает карточки товаров, включая фотографии (URL из Storage).
5.	Администратор заходит на https://username.github.io/admin-pwa/. Входит через Supabase Auth. Система проверяет роль admin (хранится в таблице profiles).
6.	Администратор через защищённые формы создаёт/редактирует заведение (координаты заполняются вручную или через клик на карте-виджете). Загружает фотографии – они отправляются в Supabase Storage, возвращённый URL сохраняется в поле image_url таблицы menu_items.

3. Детальная схема базы данных (PostgreSQL)
Таблица profiles (расширение auth.users)
sql
CREATE TABLE public.profiles (
  id UUID REFERENCES auth.users(id) PRIMARY KEY,
  role TEXT NOT NULL DEFAULT 'user' CHECK (role IN ('user', 'admin')),
  full_name TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);
Таблица cafes
sql
CREATE TABLE public.stores (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  address TEXT NOT NULL,
  latitude DOUBLE PRECISION NOT NULL,
  longitude DOUBLE PRECISION NOT NULL,
  phone TEXT,
  working_hours TEXT,
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);
Таблица menu_items
sql
CREATE TABLE public.menu_items (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  store_id UUID REFERENCES public.stores(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  description TEXT,
  price NUMERIC(10,2) NOT NULL,
  image_url TEXT,
  category TEXT,
  is_available BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT now()
);
Политики безопасности (RLS)
•	SELECT на cafes и menu_items – разрешён всем (без аутентификации).
•	INSERT/UPDATE/DELETE – только для пользователей с ролью admin, проверка через auth.uid() и запрос к profiles.
•	Storage bucket menu-photos – публичный на чтение, запись только для admin.

4. Технологический стек
Компонент	Технология / Сервис	Обоснование
Фронтенд (оба PWA)	React 18, TypeScript, Vite	Высокая производительность, компонентный подход, огромное сообщество.
Роутинг	React Router v6 (с HashRouter или кастомный 404.html для GitHub Pages)	Обеспечивает SPA-маршрутизацию на статическом хостинге.
Управление состоянием / API	Redux Toolkit + RTK Query	Удобное кеширование, автоматическая генерация хуков для запросов к Supabase.
Стилизация	Tailwind CSS	Быстрая, утилитарная, лёгкая.
PWA	Vite PWA plugin (Workbox под капотом)	Генерация service worker, манифеста, иконок.
Бэкенд	Supabase (Cloud)	Готовая БД, аутентификация, API, файловое хранилище – бесплатный тариф.
Карты	Яндекс Карты JS API 3.0	Бесплатно для публичных проектов в РФ, 25 тыс. запросов/сутки.
Хостинг фронтенда	GitHub Pages	Полностью бесплатный, CDN, HTTPS, интеграция с GitHub Actions.
CI/CD	GitHub Actions	Автоматический деплой на GitHub Pages при пуше в main ветку.
Геокодинг (опционально)	Supabase Edge Functions + Яндекс Геокодер	Серверный вызов для защиты ключа; бесплатно в рамках лимитов Яндекса.

5. Потоки данных (ключевые сценарии)
5.1. Авторизация администратора
1.	Админ открывает админ-панель → форма входа (email/password).
2.	@supabase/supabase-js отправляет запрос к Supabase Auth.
3.	Supabase возвращает JWT-токен, в app_metadata или через запрос к profiles определяется роль admin.
4.	Приложение сохраняет токен, последующие запросы к API отправляются с заголовком Authorization: Bearer <token>.
5.	RLS-политики пропускают запросы админа.
5.2. Добавление нового заведения (админ)
1.	Админ заполняет форму: название, адрес, телефон, часы работы.
2.	Получение координат:
o	Ручной ввод lat/lon или
o	Виджет Яндекс.Карты (клик по карте в админке) или
o	Вызов Edge Function geocode с адресом → ответ с координатами.
3.	Отправка POST-запроса к /rest/v1/stores с JWT.
4.	Supabase RLS проверяет роль и вставляет запись.
5.	Возвращается 201 Created.
5.3. Загрузка фотографии позиции меню
1.	Админ выбирает файл → загружает в Supabase Storage через supabase.storage.from('menu-photos').upload().
2.	Storage возвращает публичный URL вида https://.../storage/v1/object/public/menu-photos/filename.jpg.
3.	Админ заполняет остальные поля (название, цену) и отправляет POST в /rest/v1/menu_items с полем image_url.
4.	Клиентское PWA использует этот URL для показа картинки.
5.4. Отображение карты у пользователя
1.	Клиентское PWA загружается, Service Worker кеширует статику.
2.	Выполняется запрос к /rest/v1/stores?select=id,name,latitude,longitude,address. RTK Query кеширует ответ на 5 минут.
3.	Динамически подключается скрипт Яндекс.Карт (с API-ключом, ограниченным доменом GitHub Pages).
4.	После загрузки карты создаются маркеры: для каждого заведение new ymaps.Placemark([lat, lon], {balloonContentHeader: name}).
5.	При клике на маркер – загружается /rest/v1/menu_items?store_id=eq.<id> и показывается боковая панель с карточками.
5.5. Офлайн-режим (Service Worker)
•	При первом посещении кешируются: index.html, JS-бандлы, CSS, иконки.
•	Стратегия Stale-While-Revalidate для API-запросов: сначала отдаётся кеш, затем обновляется фоном, но из-за хэшей файлов при обновлении приложения нужно аккуратно управлять кешем.
•	При отсутствии сети – показывается страница "Вы офлайн", карта недоступна (требует соединения), но можно просмотреть ранее загруженное меню.

6. Развёртывание и CI/CD
6.1. Репозитории и ветки
•	Один репозиторий для клиента и админки (папки /client и /admin) или два отдельных. Для простоты деплоя на GitHub Pages проще два отдельных репозитория: cafe-client и cafe-admin.
6.2. Настройка GitHub Actions (для клиента, аналогично для админки)
Файл .github/workflows/deploy.yml:
name: Deploy to GitHub Pages

on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run build
      - uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./dist
          publish_branch: gh-pages
6.3. Настройка роутинга для SPA на GitHub Pages
Вариант 1 (рекомендованный для PWA):
•	В корень сборки (папка dist) добавить 404.html с содержимым:
<!DOCTYPE html>
<html>
<head><script>sessionStorage.redirect = location.pathname; location.replace('/');</script></head>
</html>
•	В index.html добавить скрипт, который проверяет sessionStorage.redirect и перенаправляет на нужный маршрут после загрузки React.
•	В Vite плагин vite-plugin-github-pages-spa автоматизирует это.
Вариант 2 (проще): использовать HashRouter вместо BrowserRouter. Требует изменения кода, но надёжно.
Важно: установить base: '/client-repo-name/' в vite.config.ts и basename в роутере.
6.4. Переменные окружения
В сборку через Vite передаются:
•	VITE_SUPABASE_URL
•	VITE_SUPABASE_ANON_KEY
•	VITE_YANDEX_MAPS_API_KEY
API-ключ Яндекса должен быть ограничен доменом username.github.io в кабинете разработчика.

7. Безопасность и ограничения
•	JWT-токены хранятся в памяти или sessionStorage. При обновлении страницы сессия теряется (можно сохранять в sessionStorage для удобства).
•	RLS в Supabase – единственная защита данных. Никакие запросы к таблицам не разрешены без политик.
•	CORS: Supabase по умолчанию разрешает все источники, но для GitHub Pages не требуется отдельных настроек.
•	Ограничения бесплатных сервисов:
o	Supabase: 500 МБ БД, 1 ГБ Storage, 2 ГБ трафика в месяц.
o	Яндекс Карты: 25 000 запросов/сутки.
o	GitHub Pages: 1 ГБ хранилища, 100 ГБ трафика в месяц.
•	Проверка PWA: Lighthouse (вкладка PWA) должна показывать "can be installed".

8. Заключение
Представленная архитектура позволяет запустить полностью работающее приложение для отображения заведений на карте с администрированием, не тратя деньги на инфраструктуру. Все выбранные технологии и сервисы имеют щедрые бесплатные тарифы, а GitHub Page и GitHub Actions автоматизируют развёртывание для двух PWA. Система готова к масштабированию: при росте нагрузки можно перейти на платные подписки Supabase и перенести фронтенд на Vercel, но начальный вариант уже полноценен.
Дальнейшие шаги для реализации:
1.	Создать проекты в Supabase и GitHub.
2.	Настроить таблицы и RLS (можно через SQL-скрипты из раздела 3).
3.	Сгенерировать API-ключ Яндекс.Карт, ограничить доменом.
4.	Разработать клиентское и админское PWA по описанному стеку.
5.	Настроить GitHub Actions (шаблон выше) и деплой.
6.	Проверить работу PWA, офлайн-режим, установку на устройство.
Если потребуется уточнить какой-либо раздел или получить фрагменты кода (например, инициализацию Яндекс.Карт в React с RTK Query) – дайте знать.
